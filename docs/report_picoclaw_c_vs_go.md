# PicoClaw C vs. PicoClaw Go 技術比較報告

**日期：** 2026-04-03
**專案：** ALINX AXU15EG (Zynq UltraScale+ ZU15EG) — Discord AI 助理控制 FPGA
**目的：** 評估 PicoClaw C 重寫版本相對於原始 Go 版本的效益，協助決策者判斷是否採用

---

## 摘要

PicoClaw 是一套運行於嵌入式 Linux 開發板上的 AI 助理系統，透過 Discord 接收使用者指令，呼叫雲端 LLM（Claude API），並執行 FPGA 硬體控制（LED 開關）。

原始版本以 Go 語言開發（Sipeed 開源專案），功能豐富但資源消耗大。C 語言重寫版僅保留實際使用的核心功能，大幅降低資源需求，使系統可移植至記憶體受限的嵌入式平台。

---

## 一、核心指標對比

| 指標 | PicoClaw Go | PicoClaw C | 改善倍數 |
|------|------------|------------|---------|
| 執行檔大小 | 22 MB | **964 KB** | **23 倍** |
| 記憶體用量（閒置 RSS (Resident Set Size, 常駐記憶體大小)） | ~8 MB | **1.7 MB** | **4.7 倍** |
| 記憶體用量（負載 RSS） | 50-80 MB | **3-5 MB** | **10-26 倍** |
| 虛擬記憶體（VSZ） | 1,252 MB | **2.3 MB** | **544 倍** |
| 原始碼行數 | ~70,000 行 | **~1,130 行** | 62 倍 |
| 原始碼檔案數 | 499 個 | **8 個** | 62 倍 |
| 執行緒數 | ~10（goroutine） | **2** | 5 倍 |
| 啟動時間 | ~3 秒 | **< 1 秒** | 3 倍 |

## 三、128 MB 記憶體平台可行性

本專案需評估是否可移植至僅 128 MB SDRAM 的開發板。

### 系統記憶體分配（以 128 MB 為基準）

| 元件 | 佔用記憶體 | 說明 |
|------|----------|------|
| Linux 核心 + 核心記憶體 | ~35 MB | 包含 slab、驅動程式 |
| OpenWrt 使用者空間 | ~18 MB | procd、dnsmasq、dropbear、netifd、LuCI |
| PicoClaw | 見下方 | 依版本不同 |
| **合計可用** | **128 MB** | |

### Go 版 vs C 版在 128 MB 平台的表現

| 情境 | Go 版（剩餘記憶體） | C 版（剩餘記憶體） |
|------|-------------------|-------------------|
| 閒置狀態 | 128 - 35 - 18 - 8 = **67 MB** ✓ | 128 - 35 - 18 - 1.7 = **73 MB** ✓ |
| 處理 Discord 訊息 | 128 - 35 - 18 - 50 = **25 MB** ⚠️ | 128 - 35 - 18 - 3 = **72 MB** ✓ |
| 大型 LLM 回應 | 128 - 35 - 18 - 80 = **-5 MB ❌ OOM** | 128 - 35 - 18 - 5 = **70 MB** ✓ |
| 長時間運行（數小時） | RSS 逐漸增長 → **OOM 風險** ❌ | 穩定回到 1.7 MB ✓ |

**結論：** Go 版在 128 MB 平台上閒置可運行，但處理訊息時有 OOM（記憶體不足）風險。C 版在任何情境下都安全運行，剩餘 70+ MB。

## 四、技術架構差異

### Go 版架構

```
Discord (discordgo 函式庫)
  → Channel Manager (17 種頻道抽象層)
    → Message Bus (pub/sub 事件匯流排)
      → Agent Registry (多 Agent 管理)
        → Agent Loop (goroutine 並行處理)
          → Provider Factory (20+ 供應商工廠模式)
            → Tool Registry (15+ 工具註冊表)
              → Hook System (觀察者/攔截器)
```

- 30 個 Go 套件、499 個檔案
- 60+ 個外部相依套件（Discord、Telegram、Slack、AWS SDK 等）
- 複雜的抽象層設計，適合通用平台

### C 版架構

```
Discord (手動 WebSocket + libcurl TLS)
  → Agent Loop (單執行緒處理)
    → LLM Client (libcurl SSE 串流)
      → Tool Exec (fork/exec/pipe)
```

- 8 個 C 檔案、~1,130 行
- 3 個外部函式庫（libcurl、mbedTLS、cJSON — 全部靜態連結）
- 直接、扁平的架構，專為嵌入式設計

## 五、建置與部署差異

| 項目 | Go 版 | C 版 |
|------|------|------|
| 建置工具 | Go 1.26 + make | OpenWrt SDK (gcc 12.3) |
| 交叉編譯指令 | `CGO_ENABLED=0 GOOS=linux GOARCH=arm64 make` | `aarch64-openwrt-linux-gcc -static ...` |
| 外部相依 | 60+ Go modules（自動下載） | 3 個靜態函式庫（需手動建置） |
| 建置時間 | ~30 秒 | ~10 秒 |
| 部署方式 | scp 單一執行檔 | scp 單一執行檔 |
| 目標路徑 | `/opt/picoclaw/picoclaw` | `/opt/picoclaw/picoclaw_c` |
| 設定檔 | `/root/.picoclaw/config.json` | 相同（完全相容） |

## 六、風險評估

### 採用 C 版的風險

| 風險 | 等級 | 緩解措施 |
|------|------|---------|
| 記憶體安全（緩衝區溢位） | 低 | 程式碼僅 1,130 行；使用固定大小緩衝區；複雜操作（TLS、HTTP、JSON）由成熟函式庫處理 |
| 維護成本 | 低 | 功能穩定後極少需要修改；cJSON/libcurl 由社群維護 |
| 新功能擴展 | 中 | 如需新增頻道或 LLM 供應商需手動實作；但目前需求單一 |
| Discord API 變更 | 低 | Gateway v10 為穩定版本；必要時修改 discord.c 即可 |

### 繼續使用 Go 版的風險

| 風險 | 等級 | 說明 |
|------|------|------|
| 128 MB 平台 OOM | 高 | 負載時 RSS 可達 50-80 MB，超出可用記憶體 |
| 長時間運行記憶體增長 | 中 | Go GC 不一定即時釋放記憶體 |
| 儲存空間浪費 | 低 | 22 MB 執行檔在小型 Flash 上佔比過高 |

---

## 七、建議

### 適合採用 C 版的場景

- ✓ 目標平台記憶體 ≤ 256 MB
- ✓ 僅需 Discord + 單一 LLM API + 指令執行
- ✓ 追求長時間穩定運行（無 GC 導致的記憶體波動）
- ✓ 需要最小化執行檔大小（Flash 空間受限）
- ✓ 量產嵌入式產品部署

### 適合繼續使用 Go 版的場景

- ✓ 需要多種通訊頻道（Telegram、Slack 等）
- ✓ 需要多種 LLM 供應商切換
- ✓ 需要進階功能（多 Agent、技能市場、MCP 整合）
- ✓ 目標平台記憶體充足（≥ 512 MB）
- ✓ 需要快速新增功能（Go 生態系成熟）

---

## 八、結論

| 決策維度 | Go 版 | C 版 | 勝出 |
|---------|------|------|------|
| 記憶體效率 | 8-80 MB | 1.7-5 MB | **C** |
| 執行檔大小 | 22 MB | 964 KB | **C** |
| 128 MB 平台可行性 | 有風險 | 安全 | **C** |
| 功能完整度 | 豐富（17 頻道、20+ 供應商） | 精簡（Discord + 1 供應商） | Go |
| 擴展彈性 | 高 | 低 | Go |
| 長時間穩定性 | GC 記憶體波動 | 確定性記憶體管理 | **C** |
| 本專案實際需求 | 95% 功能未使用 | 100% 功能已覆蓋 | **C** |

**建議：** 針對本專案（AXU15EG FPGA 開發板 + Discord LED 控制），以及未來移植至 128 MB 記憶體平台的需求，**建議採用 PicoClaw C 版本**。它以 23 倍更小的執行檔、10-26 倍更低的記憶體消耗，實現了完全相同的功能。Go 版本可保留作為備援方案。
